---
title: "ADR-005: Analytics on Read"
description: Compute Analytics On Read, Not On Write.
---

**Status:** Accepted — 2026-06-11

## Context

Two performance budgets pull in different directions:

- **ASR-03 / NF-PERF-01**: submission requests must respond within 500 ms
  (p90) under 40 concurrent students per session. The submission hot path
  (`POST /api/student/session/:id/submit`) must therefore do as little
  work as possible beyond persisting the submission, computing
  correctness, first-correct status, and score (F-SUB-03 to F-SUB-05).
- **ASR-10 / NF-PERF-03**: teacher analytics (student/group leaderboards,
  per-question accuracy and option distribution, CBC competency
  indicators — F-ANA-01 to F-ANA-04) must be available within 2 seconds
  (p90) for a class of 40 students and 20 questions.

A naive design might update leaderboard totals, per-question tallies, and
CBC indicators incrementally on every submission — keeping analytics
"pre-computed" but adding non-trivial computation to the submission hot
path, directly threatening ASR-03 as the number of students/questions
grows.

## Decision

**Submission processing (hot path)** does the minimum required by F-SUB-01
to F-SUB-06: validate the reasoning-length gate, persist the submission,
determine `is_correct` and `is_first_correct` by checking against
`correct_answer` and any prior submissions for that question, compute the
score using the fixed formula, and award a badge if applicable. No
leaderboard, aggregate, or CBC computation happens here.

**Analytics (read path)** — `GET /api/teacher/sessions/:id/analytics` and
`GET /api/teacher/sessions/:id/live` — compute leaderboards, per-question
statistics, and CBC indicators **on demand**, by reading the relevant
`submissions` records for the session and aggregating in memory at request
time.

This is an explicit "compute on read" strategy: the cost of aggregation is
paid by the (less frequent, teacher-facing) analytics request, not by the
(frequent, student-facing) submission request.

## Consequences

### Positive

- The submission hot path remains O(1) relative to the number of other
  students/questions in the session (aside from the "first correct" check,
  which is a lookup against existing submissions for the *same* question
  only) — directly protecting ASR-03 even as session size grows toward
  40 students.
- Analytics logic is centralised in one place (the analytics route
  handler), making it straightforward to adjust the CBC indicator formulas
  (F-ANA-04) without touching the submission path or risking regressions
  in scoring (F-SUB-05).
- This keeps [ADR-004](/adrs/adr-004/)'s repository abstraction simple: analytics reads a
  collection and aggregates in application code, rather than requiring
  incremental-update logic to be duplicated across every write path that
  could affect an aggregate.

### Negative

- For a class of 40 students and 20 questions (800 submission records),
  the analytics endpoint must read and aggregate up to 800 records per
  request. This is the basis for the ASR-10 budget (2 seconds, p90) and
  must be load-tested at this scale before release.
- If analytics is requested very frequently during a live session (e.g. a
  teacher leaves the live monitor open with auto-refresh), repeated
  aggregation over the same growing dataset could become wasteful. The
  live monitor (F-MON-01 to F-MON-03) uses a lighter-weight, current-
  question-only aggregation (not the full session analytics) specifically
  to avoid this.

## Revisit Triggers

- If load testing shows the 800-record aggregation exceeds the 2-second
  budget (ASR-10) at the target scale, introduce incremental aggregate
  tables (e.g. updated within the same transaction as the submission, once
  PostgreSQL transactions are available per ADR-004) for the specific
  aggregates that are too slow to compute on read.
- If "live" analytics views are requested at a frequency that makes
  per-request aggregation costly even for the lighter live-monitor
  payload, consider caching the aggregation for a short TTL (e.g. 2–3
  seconds) bounded by the ASR-02 real-time budget.

title: "ADR-004: Repository Pattern & Question Schema"
description: Repository Pattern for Data Access and Generic Question Schema.
---

**Status:** Accepted — 2026-06-11

## Context

Gillytech v1 uses **lowdb** (a single JSON file) as its data store, chosen
for simplicity during the pilot phase (SRS §2.4). However, the SRS
explicitly constrains this:

> "The initial data layer (lowdb/JSON) must not leak into the API
> contract — all data access goes through a repository-style module
> boundary to ease a future PostgreSQL migration." (SRS §2.5, ASR-04)

Separately, the system must support adding new question types (currently
multiple-choice, with true/false anticipated) without changing the
submission API contract or WebSocket protocol (ASR-08).

## Decision

### 1. Repository-style data access

All reads and writes to the underlying store go through `db/database.js`,
which exposes a single `db` object (`db.data.{users, classes, sessions,
questions, submissions, ...}`) with `db.read()` / `db.write()` methods.
Route handlers and the WebSocket layer interact **only** with this module
— no route handler opens a file or constructs a query string directly.

This module is the *only* place that knows the storage technology is
lowdb. A future migration to PostgreSQL replaces the internals of
`db/database.js` (and the read/write calls within route handlers, which
follow a consistent `await db.read(); ... ; await db.write();` pattern)
with equivalent ORM/query calls, **without changing**:

- HTTP request/response shapes,
- WebSocket message shapes,
- the frontend `api.js` client.

### 2. Generic question schema

Questions are stored with a `type` discriminator and a generic
`content_json` field:

```json
{
  "id": "...",
  "session_id": "...",
  "order_index": 0,
  "type": "multiple_choice",
  "marks": 10,
  "time_limit_sec": 120,
  "content_json": { "text": "...", "options": ["...", "...", "...", "..."] },
  "correct_answer": "2"
}
```

Server-side scoring logic (in `routes/student.js`) dispatches on `type` to
determine how `answer` is compared against `correct_answer`. For
`multiple_choice`, this is a string equality check on the option index.
Adding `true_false` (already referenced in the session builder UI) requires
only a new branch in the scoring function and a new option set
(`["True", "False"]`) — the submission endpoint, request shape, and
response shape are unchanged.

## Consequences

### Positive

- The lowdb → PostgreSQL migration path (a known future need, SRS §2.4) is
  isolated to one module, satisfying ASR-04. This was validated during
  initial development: the seed script, all route handlers, and the
  WebSocket layer interact with `db.data.*` collections in a way that maps
  directly to relational tables (e.g. `db.data.submissions` →
  `submissions` table).
- The generic question schema satisfies ASR-08: the `content_json` /
  `type` pattern means new question types are additive (new `type` values
  and scoring branches) rather than requiring schema migrations to the
  submission or session APIs.

### Negative

- lowdb provides no transactional guarantees; concurrent writes (e.g. two
  students submitting at nearly the same time) rely on Node's
  single-threaded event loop and the `await db.read(); ...; await
  db.write()` pattern being effectively atomic per request. This is
  acceptable at pilot scale but must be re-examined when migrating to
  PostgreSQL (where proper transactions should be used for
  read-modify-write sequences such as "check if first correct, then
  award badge").
- The generic `content_json` field is untyped from the database's
  perspective; validation of question content shape must happen in
  application code (route handlers / session builder) rather than via
  schema constraints. This is a deliberate trade-off favouring
  extensibility (ASR-08) over compile-time/schema-time safety.

## Revisit Triggers

- If ASR-12 (aggregated interaction telemetry — see
  [ASR Mapping](/requirements/asr/#notes-on-usability--interaction-telemetry))
  is approved after data-minimisation review, add the telemetry fields as
  additive properties on the `submissions` record (e.g.
  `time_to_first_keystroke_ms`, `total_reasoning_duration_ms`,
  `backspace_count`) rather than a new collection — consistent with this
  ADR's existing pattern of extending records additively (as already done
  for the `content_json`/`type` question schema).
- When migrating to PostgreSQL: introduce explicit transactions for the
  "first correct" check-and-award sequence (F-SUB-04, F-SUB-06) to avoid
  race conditions under concurrent submission load (ASR-03, ASR-07).
- If question types diverge significantly in scoring complexity (e.g.
  free-text with AI-assisted grading, per the planned reasoning
  evaluator — see [Architectural Style](/architecture/as/)), consider extracting a
  pluggable "scoring strategy" module rather than a single dispatch
  function.

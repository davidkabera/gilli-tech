title: "ADR-001: Modular Monolith"
description: Adopt a Modular Monolith for the Gillytech Backend.
---

**Status:** Accepted — 2026-06-11

## Context

Gillytech requires a backend that serves:

- A REST API for authentication, teacher session/question management,
  student submissions, and analytics (Section 4 of the SRS).
- A WebSocket server for real-time chat, presence, and session/question
  advancement events (ASR-02, ASR-06, ASR-07).

The system must support real-time updates within 5 seconds (p90)
across multiple concurrent sessions (ASR-07), while remaining simple
enough for a small team to build, test, and operate for a single-school
pilot (SRS §2.4, §5.6 Availability — "no formal SLA for v1").

Two broad options were considered:

1. **Microservices** — separate services for auth, sessions, submissions,
   chat/WebSocket, and analytics, communicating over a message bus or
   internal APIs.
2. **Modular monolith** — a single Node.js process with clearly separated
   modules (routes, WebSocket layer, repositories) sharing an in-process
   broadcast registry.

## Decision

We will build Gillytech as a **modular monolith**: a single Express
application process that hosts both the REST routes and the WebSocket
server (`ws`), sharing the same data-access (repository) layer and an
in-process connection registry for broadcasting.

Internal modules are separated by responsibility:

- `routes/auth`, `routes/teacher`, `routes/student`, `routes/admin` — REST
  route handlers, thin and delegating to repositories.
- `websocket.js` — WebSocket upgrade handling, connection registry
  (`sessionId → groupId → connections`), and broadcast helpers.
- `db/` — repository-style data access (see [ADR-004](/adrs/adr-004/)).
- `middleware/auth.js` — shared JWT verification and role-based guards,
  applied to both REST routes and the WebSocket handshake.

## Consequences

### Positive

- A single deployable unit simplifies operations for a single-school pilot
  (no service discovery, no distributed tracing needed yet).
- Sharing an in-process connection registry between REST handlers (e.g.
  "teacher advances question") and the WebSocket broadcast layer is trivial
  — a function call, not a network hop — which directly supports the
  5-second real-time budget (ASR-02) with minimal latency overhead.
- Module boundaries (routes / websocket / repositories / middleware) keep
  the codebase navigable and testable without the operational cost of
  microservices.

### Negative

- Scaling beyond a single process (e.g. multiple Node.js instances behind
  a load balancer) requires moving the connection registry and broadcast
  mechanism to a shared layer (e.g. Redis pub/sub) — this is explicitly
  deferred (see "Revisit Triggers" below).
- A bug in one module (e.g. analytics computation) runs in the same
  process as the real-time chat path; careful module boundaries and
  testing are required to avoid one feature degrading another.

## Revisit Triggers

This decision should be revisited if:

- The platform grows beyond a single school or a single Node.js process
  can no longer hold the required concurrent WebSocket connections
  (ASR-07 assumes ~400 connections; if this grows by an order of
  magnitude, a shared broadcast layer such as Redis pub/sub becomes
  necessary).
- Distinct modules require independent scaling or deployment cadences
  (e.g. analytics becomes a heavy batch workload).

---
title: "ADR-003: WebSocket Broadcast Scoping"
description: WebSocket Broadcast Layer Scoped by Session and Group.
---

**Status:** Accepted — 2026-06-11

## Context

Real-time collaboration is central to Gillytech (SRS §4.3, §4.4):

- Group chat messages must reach only members of the sending student's
  group, plus the teacher (F-CHAT-02, F-CHAT-05; ASR-06).
- Question-advance and session-status events must reach **all**
  participants of a session (F-SES-05).
- The system must support multiple concurrent sessions without
  cross-session interference in delivery latency (ASR-07).
- All of this must happen within a 5-second (p90) delivery budget
  (ASR-02).

A naive approach — broadcasting every WebSocket message to every
connected client and filtering client-side — would violate ASR-06
(a malicious or modified client could simply not filter) and would not
scale cleanly with ASR-07.

## Decision

The WebSocket layer (`websocket.js`) maintains an **in-memory connection
registry** structured as:

```
sessionClients: Map<sessionId, Set<WebSocket>>
clientMeta:     Map<WebSocket, { userId, name, role, sessionId, groupId }>
```

On connection, the server looks up the student's `group_members` record
for the given session and stores `groupId` in `clientMeta`. Two broadcast
functions are provided:

- `broadcastGroup(sessionId, groupId, payload, exclude?)` — sends to
  connections in `sessionClients[sessionId]` whose `clientMeta.groupId`
  matches `groupId`, **or** whose `role === 'teacher'` (teachers receive
  all groups' messages, per F-CHAT-05). Used for chat and presence events.
- `broadcastToSession(sessionId, payload)` — sends to all connections in
  `sessionClients[sessionId]`, regardless of group. Used for
  `question_advance` and `session_update` events, triggered from the
  REST layer (`PATCH /api/teacher/sessions/:id`) via the shared in-process
  registry (enabled by [ADR-001](/adrs/adr-001/)).

Group scoping is determined **server-side at connection time** from the
persisted `group_members` table, never trusted from client-supplied data
— directly satisfying ASR-06.

## Consequences

### Positive

- Group confidentiality (ASR-06) is enforced structurally: a connection's
  `groupId` is fixed at connection time from server-side data and cannot
  be changed by the client.
- Session-scoped maps mean that broadcast iteration for one session only
  touches that session's connections, bounding the cost of a broadcast
  and supporting ASR-07 (multiple concurrent sessions).
- Because the registry is in-process ([ADR-001](/adrs/adr-001/)), `broadcastToSession` can
  be called directly from a REST route handler (teacher advances question)
  with no additional network hop, helping meet the 5-second budget
  (ASR-02).

### Negative

- The registry is in-memory and per-process. If the backend is scaled to
  multiple processes/instances (see ADR-001 revisit triggers), this
  registry must be replaced or augmented with a shared pub/sub mechanism
  (e.g. Redis) so that a broadcast triggered on one instance reaches
  clients connected to another instance.
- Reconnection handling (e.g. a student's browser refreshes mid-session)
  must re-establish `clientMeta` from persisted data; any transient state
  held only in `clientMeta` (none currently) would be lost on reconnect.

## Revisit Triggers

- Moving beyond a single backend process (ADR-001 revisit trigger)
  requires replacing the in-memory `Map`-based registry with a
  shared/distributed equivalent.
- If group membership can change *during* an active session (currently
  disallowed per SRS §5.5 Business Rules — "groups are not reshuffled
  mid-session"), the registry would need a mechanism to update
  `clientMeta.groupId` for already-connected clients.

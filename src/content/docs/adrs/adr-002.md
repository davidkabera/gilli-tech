title: "ADR-002: JWT Authentication"
description: Layered REST API with Shared JWT Authentication and Role-Based Authorisation.
---

**Status:** Accepted — 2026-06-11

## Context

Gillytech has three user roles (student, teacher, admin) accessing
overlapping but distinct sets of REST endpoints and a shared WebSocket
channel (SRS §2.3, §3.4). The system must:

- Enforce authentication on all endpoints except login (NF-SEC-02, ASR-05).
- Enforce role-based authorisation server-side, independent of client-side
  routing (NF-SEC-03, ASR-05).
- Never store credentials in plaintext (NF-SEC-01, ASR-09).
- Apply the same identity to both REST requests and the WebSocket
  connection used for chat (ASR-05, ASR-06).

## Decision

We will use **JSON Web Tokens (JWT)** as the single credential format for
both REST and WebSocket authentication, validated by a shared middleware
module (`middleware/auth.js`).

- On login, the server verifies the submitted password against a
  **bcrypt** hash (`bcryptjs`) and issues a signed JWT containing
  `{ id, role, name }` with a 24-hour expiry.
- REST routes apply an `auth` middleware that verifies the JWT from the
  `Authorization: Bearer <token>` header and attaches `req.user`.
- Role-specific routers (`teacher`, `student`, `admin`) additionally apply
  a `requireRole(...)` middleware that checks `req.user.role` against an
  allow-list, returning HTTP 403 for mismatches.
- The WebSocket upgrade handler extracts the same JWT from a `token` query
  parameter, verifies it with the identical secret and library, and closes
  the connection immediately if verification fails.

This gives a single source of truth for "who is this request from, and
what are they allowed to do" across both transport mechanisms.

## Consequences

### Positive

- One auth implementation to test and audit, reducing the chance of an
  inconsistency between REST and WebSocket authorisation (directly
  supports ASR-05, ASR-06).
- Stateless verification (no server-side session store) keeps the modular
  monolith ([ADR-001](/adrs/adr-001/)) simple to scale horizontally later, since any
  instance holding the JWT secret can verify a token.
- bcrypt hashing satisfies NF-SEC-01 / ASR-09 with a well-understood,
  audited library.

### Negative

- JWTs cannot be revoked before expiry without an additional denylist
  mechanism; a 24-hour expiry bounds this risk for the v1 pilot but should
  be revisited if immediate revocation (e.g. on account compromise)
  becomes a requirement.
- The JWT secret becomes a single high-value credential; it must be
  provisioned via environment configuration, never committed to source
  control.

## Revisit Triggers

- If the platform requires immediate session revocation (e.g.
  administrator "force logout"), introduce a token denylist or move to
  opaque session tokens with server-side lookup.
- If horizontal scaling introduces multiple JWT-signing instances,
  ensure the signing secret is managed via shared, secure configuration
  (not generated per-instance).
